# 矩阵交叉检查 Skill

## 用途

本Skill用于对代码中的多数据结构进行系统性交叉检查，通过构建矩阵来发现：
- 字段缺失/未同步
- 传参遗漏
- 引用解析问题
- 数据流转路径中的断点

## 触发场景

1. 添加复杂大功能，需要与多个旧数据结构关联时
2. 用户明确说"对xx和xx进行深度检查"
3. 用户说"构建矩阵检查xx"
4. 重构前需要确认数据完整性
5. 发现数据丢失/不同步bug后排查根因

## 检查流程

### 步骤1：确定检查范围

明确需要检查的数据结构/类/模块：
- 列出所有相关类名
- 明确数据流转的起点和终点
- 确定需要检查的维度（如：ExposeData、DeepCopy、传参等）

### 步骤2：收集字段清单

对每个类，收集：
```
类名:
- 字段1: 类型, 用途
- 字段2: 类型, 用途
- ...
```

### 步骤3：构建检查矩阵

根据检查目标选择矩阵维度：

**持久化检查矩阵**（RimWorld项目常用）：
| 字段 | ExposeData() | DeepCopy() | ResolveReferences() | ResetToDefault() |

**数据流转检查矩阵**：
| 字段 | 来源A | 来源B | 目标X | 目标Y |

**传参检查矩阵**：
| 方法 | 参数1 | 参数2 | 调用者A | 调用者B |

### 步骤4：填充矩阵并标记问题

- ✅ 已覆盖/已传递
- ❌ 未覆盖/未传递
- ⚠️ 部分覆盖/有条件覆盖
- 🔴 发现bug/逻辑错误

### 步骤5：生成问题清单

按优先级分类：
- 🔴 致命：导致功能完全失效
- 🟡 中等：可能导致数据不一致
- 🟢 低：代码风格/潜在风险

### 步骤6：输出检查报告

包含：
1. 矩阵表格
2. 问题清单（带位置链接）
3. 修复建议
4. 风险评估

## 常见检查模式

### 模式A：IExposable数据类完整性检查

检查点：
1. 所有字段是否都在 `ExposeData()` 中处理
2. 引用类型（Def）是否用defName存储
3. 缓存字段是否标记 `[Unsaved]`
4. `DeepCopy()` 是否覆盖所有字段
5. `ResolveReferences()` 是否解析所有引用
6. `ResetToDefault()` 是否重置所有可变字段

### 模式B：预设/配置系统检查

检查点：
1. 预设保存时所有字段是否都被保存
2. 预设加载后引用是否重新解析
3. 全局设置与存档级别数据同步是否完整
4. 导出/导入后数据完整性

### 模式C：数据流转路径检查

检查点：
1. 起点到终点的每个中间节点
2. 每个转换/拷贝操作
3. 异步/延迟加载场景

## 输出格式

```markdown
# 矩阵检查报告

## 一、字段清单

### 类名
| 字段 | 类型 | 说明 |
|-----|------|------|
| field1 | string | ... |

## 二、检查矩阵

| 字段 | ExposeData | DeepCopy | ... |
|-----|-----------|----------|-----|
| field1 | ✅ | ✅ | ... |
| field2 | ❌ | ✅ | ... |

## 三、问题清单

### 🔴 致命
1. **问题描述**
   - 位置：`文件.cs:行号`
   - 影响：...
   - 修复：...

## 四、修复建议
...
```

## 注意事项

1. **不要假设**：每个字段都要实际检查代码确认
2. **关注引用类型**：Def引用、缓存字段容易出问题
3. **检查集合类型**：List/Dictionary的深拷贝是否正确
4. **验证调用链**：数据实际流转路径可能与预期不同
5. **标记Unsaved字段**：确认是否真的不需要持久化

## 示例

用户说："帮我检查预设功能的持久化完整性"

执行：
1. 找出所有与预设相关的类（FactionGearPreset, FactionGearData, KindGearData等）
2. 构建字段清单
3. 创建矩阵：字段 × (ExposeData | DeepCopy | ResolveReferences)
4. 填充并标记问题
5. 输出报告
