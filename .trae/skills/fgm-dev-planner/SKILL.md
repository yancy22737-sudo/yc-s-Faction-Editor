# FGM 开发方法规划

本技能用于把需求拆解为符合本项目规范的实现方案：模块边界、文件落点、数据流、UI 组件拆分、测试与发布流程。

## 何时调用

- 用户确认了功能方向，需要具体的实现步骤与落地路径
- 需要重构但要保持低风险，要求“先拆分模块、再逐步替换”
- 需要同时考虑：UI 组件化、业务逻辑分层、语言适配、版本号与更新日志

## 输出要求（必须包含）

1) 目标与非目标
- 目标：可验证的行为描述
- 非目标：明确不做什么，避免范围蔓延

2) 模块设计
- UI：按 Panels 拆分，Window 只做组装与路由
- 业务：Managers 负责流程编排，Data 只存模型与规则，IO 只管读写
- Patch：只做必要的入口拦截/数据注入，避免把业务逻辑塞进 Patch

3) 数据流与状态
- 状态归属：UI 状态集中到 UI/State（例如会话对象），禁止大量静态字段
- 持久化：明确写入位置与版本迁移策略（若涉及）

4) 语言与 UI 适配
- 新增字符串：必须同时落到中英 Strings
- UI 布局：考虑不同语言长度与缩放，避免硬编码宽度导致截断

5) 版本与日志
- 版本号：每次 build 前按 x.y.z 升级
- 更新日志：同步更新 VersionLog.txt

6) 验证与回归
- 最小回归路径：列出 3-6 个用户操作路径
- 边界条件：空数据、异常输入、极端数量、撤销/重做（若相关）
- 手工测试清单：优先复用现有 MANUAL_TEST_CHECKLIST/TEST_CASES

## 推荐的拆解方式

- 先落“数据模型与规则”再落“UI 操作”
- 每个步骤都应能独立编译与运行，避免一次性大改
- 新增能力优先走扩展点，而不是修改大量既有代码

## 交付物模板（建议直接按此输出）

- 方案概览：一段话总结实现路径
- 文件/模块变更：按目录列出要新增/改动的点（不写到变量级别）
- 关键实现点：3-8 条，覆盖入口、校验、数据落点、UI 触发、撤销/持久化
- 测试与发布：怎么验证 + 需要更新的版本/日志/语言文件
